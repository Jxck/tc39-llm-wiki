## AsyncContext yield\*

- Slides: [link](https://docs.google.com/presentation/d/1g7Xgf9uAxv5gZYvms23m2_hP513IRYNktz0fDiE1NmA)
  NRO が generator の `yield*` における AsyncContext の伝播挙動の変更を提案した。現状は `yield*` のループ内で毎回コンテキストを復元しているが、これをループ終了後の1回だけに変更することで、呼び出し元の `.next()` 呼び出し時点のコンテキストを内側の iterator に伝播できるようにする。これにより、ユーザーランドで `AbortSignal` 等のコンテキストをマージするヘルパーが書けるようになる。
  MM からの確認質問(コンテキストの結合自体は提供しないこと、reify するだけであること)に NRO が回答し、変更内容への異論は出なかった。
  Stage 2 の提案だが、spec text はほぼ安定しており、web 統合待ちのため Stage 2.7 に近い状態にあることが共有された。

## Update on proposal-await-dictionary

- Slides: [link](https://docs.google.com/presentation/d/1TaLCZt2jJtrVY1PjFd49jlBW-vAVil4NYsOzAyYPP7Q/)
  ACE が `Promise.all` の代替として、名前付きオブジェクトを受け取り対応する名前で結果を返す `Promise.allKeyed`(命名は `Iterator.zip` / `Iterator.zipKeyed` に倣う)を改めて紹介し、consensus ではなく committee の感触を確認する場として発表した。
  質疑では `Promise.allSettledKeyed` の追加についてKG・ZTZ・MM から支持が集まり、`all` と `allSettled` はどちらも必要(互いに代替不可)という MM の指摘もあり、追加メソッドとして採用する方向に。シンタックス拡張については MM・OFR からの議論の末、別提案として将来検討することとし、本提案には含めない方針となった。JSL からは「Stage 2.7 を求めるのは時期尚早」との意見があり、ACE もこれに同意。レビュワーとして JHD・JSL が立候補した。

## Import Bytes for stage 2.7

- Slides: [link](https://slides-import-bytes-2025-09.vercel.app/)
  STY が Import Bytes 提案(`type: "bytes"` で任意のバイト列を `Uint8Array`(immutable `ArrayBuffer` 裏付き)として import する仕組み)の Stage 2.7 を要求した。前回から `type: "buffer"` → `type: "bytes"` への変更、戻り値を `ArrayBuffer` ではなく `Uint8Array` にする変更などフィードバックを反映済みで、spec text は完成しレビュワー3名による review も完了している。
  MM・JSL・NRO・JHD など多数の支持表明があり、特に静的 import では同期的にバイト列を取得できる点(fetch では不可能)が利点として補足された。Immutable ArrayBuffer 提案が現在 Stage 2.7(conditional Stage 3)であるため、本提案はそれを超えて先に進めない依存関係があることも確認された。
  Stage 2.7 への consensus が得られた。

## Update on proposal-module-global

- Slides: [link](https://github.com/endojs/proposal-module-global/blob/main/slides/2025-09-stage1-update.pdf)
  KKL・ZTZ が Stage 1 の module-global(Compartment)提案の進捗を報告した。新しい global scope と module map を同一 realm 内に作る仕組みで、テスト用の隔離環境やサプライチェーン攻撃対策(LavaMoat 等)、マルチテナントなプラグインシステムが主な動機として説明された。現状はユーザーランドで `with` 文・sloppy mode・direct `eval`・Proxy を組み合わせたシムで実現しているが、パフォーマンスや忠実性に限界があり、言語サポートが必要とされる。
  前回受けたフィードバックを踏まえ、過去の Compartment 提案のアイデアを取り込んだ再設計を進めていること、新しい評価パス(eval)を主目的としないこと、module source への非シリアライズ可能なフックは追加しないことなどが共有された。KG から ShadowRealm との関係性について強い疑問が提起され、時間切れのため後続セッションへ持ち越された。

## Temporal update and normative change

- Slides: [link](https://ptomato.name/talks/tc39-2025-09/)
- 提案ページ: [Temporal](../../proposals/temporal.md)
  PFC が Temporal の進捗を報告した。現在2つの実装(V8/Boa、ともに `temporal-rs` ベース)が Test262 の 99% に合格し、Kiesel engine も新たに実装を追加した。Stage 4 への道筋として、Intl Era Month Code が次回 Stage 3 に到達すること、V8 実装の unflag、`staging` フォルダに残るテストの本体への移動が必要と説明された。Canonicalization・Era Month Code 提案と合わせて3提案同時に Stage 4 を目指す案も示された。

## How Websites are Put Together

- Slides: [link](https://docs.google.com/presentation/d/1vEYoTix5yHN3vc1cXZQCk87P8d_GqL0bYsOpoEcnjuo/edit?usp=sharing)
  KG が consensus を求めない情報提供セッションとして、大規模商用サイトが複数チーム・複数企業のスクリプトの非協調的な集合として構築されている実態を説明した。スクリプトの読み込み順は制御できず、ビルトインのパッチ適用がページのライフタイム全体で継続するため、グローバルな協調を前提とする仕組み(CSP のスクリプトハッシュ運用や、全ビルトインを freeze する「ロックダウンモード」、単一の import map など)は実用上機能しないと主張した。
  結論として KG は、committee はこの種のサイトで実際に使える(=スクリプト単体で協調なしに使える)機能に時間を割くべきだという私見を述べた。CM からは、この実態こそが Compartment 提案(モジュール間の隔離によって協調の必要性を減らす)の説得力ある論拠になっているとの指摘があった。MAH・REK・JSL・WH らとの間で、将来世代のアプリケーションへの投資をどう考えるかについて議論が続いたが、consensus を求める議題ではなく結論は出ていない。
