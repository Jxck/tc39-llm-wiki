# 2026-03-10 (113th TC39 Meeting, Day 1)

## import defer interaction with tc39/ecma262#3715, and normative PR Shield dynamic import defer against `Promise.prototype` monkey patch
- Slides: [link](https://docs.google.com/presentation/d/1T1M7X-cS_ODKOl6Nsq-_YmHusVoD6bPtAS_MHzxzgTw)
- import defer に対する 2 件の normative PR を提示。1 件目は dynamic import defer が非同期依存の待機に `PerformPromiseAll` を使うため、`Promise.prototype.then` を monkey patch すると spec 内部の top level capability promise が観測できてしまう leak を塞ぐもの。`then` を呼ばず PerformPromiseThen 相当の built-in steps を使う形に変更する(V8 実装中に発見)。
- 2 件目は前回 ECMA262 で合意された namespace re-export の normative PR と組み合わさると、deferred namespace の re-export が誤って eager 化される問題を、phase を正しく伝播させて修正するもの。
- KG は強い支持を表明し、可能なら一般化したいが piecemeal でも進めるべきとした。
- 結論: 両変更とも consensus を得た。

## Abort Protocol Discussion
- Slides: [link](https://github.com/jasnell/discussion-abort-protocol/blob/main/slides-export.pdf)
- JSL が abort/cancellation を言語レベルに入れるべきかの問題提起。エコシステムは WHATWG の AbortController/AbortSignal に収束しているが、これは言語機能でなく非対応環境もあるため、既存の解を最小限の変更で満たせる abort signaling を言語に持たせたいという問題提起(提案ではなく合意確認の discussion)。
- MM は同期コールバックを Unix signal handler になぞらえ危険視し、safe point での bit チェックを推奨。KG は web で 10 年近く実問題化していないと反論。JHD は AbortSignal が最善か疑問を呈し「task cancellation」として一般化すべきとした。
- RBN が過去の cancellation 提案(WHATWG との調整難航で頓挫)の経緯を共有し、`import()` 呼び出しや `then` 継続、shared structs など cancellation が有用な箇所を挙げた。設計論点(同期コールバック、combinators、AbortSignal との整合)は引き続き議論。
- 結論: 言語に入れるべきという点で概ね合意。JSL は新規ではなく既存の旧 cancellation 提案の champion に加わり再起動を進める。

## TypedArray concat for Stage 2
- Slides: [link](https://github.com/tc39/proposal-typedarray-concat/blob/tc39-plenary-march-2026/slides-export.pdf)
- TypedArray/ArrayBuffer/SharedArrayBuffer 向けに最適化可能な concat 静的メソッドを与える提案の Stage 2 ask。引数は iterable、length 省略可(切り詰め/zero-fill)、options bag で resizable/immutable 指定。
- 論点: 名前(`concat` は String/Node Buffer/`Iterator.concat` と衝突。`fromChunks`/`concatFrom` 等を検討)、iterable の消費順序(全消費してから検証するか、コピーしながらか。observability に影響)が主要。
- DLM が現行の problem statement を block。実装上 rope 系構造しか最適化手段がなく TypedArray/ArrayBuffer は linear のまま保つべきというのが blocking concern。KM も rope には反対(O(1) アクセスを期待)。ergonomics 目的の concat 自体には反対なしだが problem statement の文言が問題。
- 結論: problem statement の文言に異議。JSL が文言を見直し、Day 3 に再度 Stage 2 を ask する。

## Amount for Stage 2
- Slides: [link](https://docs.google.com/presentation/d/1SAD0bgB_SOWrtz1YZuySC4YHdMxBs0vTnS7buQa311c/edit?slide=id.g3cb9b164150_1_50#slide=id.g3cb9b164150_1_50)
- 数値に単位を付与する Amount の Stage 2 ask(*印付き)。旧版の mathematical value から、入力型に応じ number/bigint/decimal string を保持する設計へ刷新。CLDR の units.xml を normative reference として単位変換を再導入し、API を大幅簡素化(`.with()`・fraction/significant digits accessor 削除)。算術は number 空間で行い 402 の FormatNumericToString に依拠。
- WH が、提案 spec が会合 3 時間前に差し替わりレビュー不能だった手続き上の問題に加え、(1) US の person height で 2m を 79 inch と返す localization 上の誤り(mixed unit「foot+inch」が必要)、(2) 7 feet→inches の多重 rounding で truncate 時 83 inch になる exact conversion の誤り、を blocking 級の懸念として指摘。double/decimal の別でなく多重 rounding が原因と強調。
- 変換の精度保持(JRL「無限精度変換後に precision を増やすのは誤り」)を巡る議論。実装者(OFR・DLM)は number 空間での変換を支持。
- 結論: 今回 Stage 2 は ask せず。WH と rounding 問題を継続検討し、May に Stage 2 を ask する方針。

## Intl: Keep Trailing Zeros for Stage 3
- Slides: [link](https://docs.google.com/presentation/d/13zyaDAuriW2A4pwzklf0EZhOSUUmWqZxyIjzCcjWkXU/edit?usp=sharing)
- `Intl.NumberFormat`/`Intl.PluralRules` が digit string 入力の trailing zeros を現状無視している挙動を修正する Stage 2.7 提案。新 option 値 `trailingZeroDisplay: "stripToMinimum"` で従来挙動も選択可能。number/bigint 入力の挙動は不変。
- 残 2 issue を議論。#15: 非常に小さい値(例 `"1.2e-10"` を maxFractionDigits 5 で)が現状 `"0.0"` になる挙動の是非。NRO は `0.0` が最悪(誤った significant digits を示唆)とし bottom 出力(maxFractionDigits 準拠)か単なる `0` を支持。#17: 整数末尾ゼロの扱い(`"700"` → `7.00E2` か `7E2` か)。NRO・SFC は `7.00E2`(3 significant digits)を支持、WH は `7E2` を選好。
- EAO は spec text 変更量を理由に今回は Stage 3 を ask しない方針。
- 結論: #15 は挙動を変更する方向で解決、#17 は現行挙動(整数末尾ゼロを significant として保持)を維持。advancement は将来の会合へ。

## `JSON.parseImmutable` update
- Slides: [link](https://docs.google.com/presentation/d/1NXb3HWjt0BiVJl2SZlAVeSvZIBlzO1z9X28hJL28w7s)
- Records & Tuples 撤回に伴い、それに紐付いていた `JSON.parseImmutable` の進路を議論。選択肢は (1) 撤回、(2) composites を返す、(3) deeply `Object.freeze` した plain object/array を返す(champion 選好)。
- MF・JHD・ACE は composites 不適と表明(用途が key 中心でデータ向きでない、tuple 相当が無く JSON 配列を扱えない)。MM は realm 跨ぎの安全性を理由に composites の利点を挙げつつ態度保留。frozen object は `Object.prototype` 継承のため realm leak が起きうると指摘。
- NRO・JSC は frozen を返すなら `parseFrozen` 等へ改名を提案。MM の reviver 不要案は KG の「parse-with-source-text は reviver 前提」で撤回。KG は別名でなく options bag 案を提示。
- 結論: Records & Tuples 撤回後も有用との感触で option 3(deeply frozen)方向へ。名前・reviver 挙動・Stage 1 への差し戻し要否は未決(offline 継続)。

## TypedArray find within for Stage 2
- Slides: [link](https://github.com/tc39/proposal-typedarray-findwithin/blob/tc39-plenary-march-2026/slides-export.pdf)
- TypedArray 内の部分列を探す提案の Stage 2 ask。現状 `search`/`searchLast`/`contains` の 3 メソッド、needle は TypedArray と iterable(string は throw)。
- WH は iterable needle が型不一致(例: 64-bit int array を number needle で、Float32Array を `1.1` で検索)で silent に見つからない footgun を指摘し、needle を haystack 型に coerce すべきとした。KM は iterable は要素ごとのオブジェクト割当でかえって高コストと指摘し、KG は iterable 引数を一旦削除する案に同意。
- `contains` 不要論(`search` の -1 で代替可、名前が誤解を招く)、wrong-type で throw か -1 か、SharedArrayBuffer 対応可否(needle snapshot の要否)を議論。RBN/RGN は -1 派、SAB の footgun は既に許容済みとの整理。
- 結論: Stage 2 には至らず(`contains` 削除・iterable needle 削除・SAB は snapshot せず保持、の変更が必要)。MF の提案で conditional でなく Day 3 の overflow slot で再 ask する方針(Day 3 へ継続)。

## Iterator Includes for Stage 1, 2, or 2.7
- Slides: [link](https://docs.google.com/presentation/d/1a-1RQayP-tcJd2VAhFGoiL_dRT8gD8_qrp__qSgC1A4)
- iterable に対し値の有無を調べる `Iterator.prototype.includes` の新規提案(`Array.prototype.includes` と同じ動機、SameValueZero 比較が手書きしにくい点が JS 固有の追加動機)。KG が「iterator でなく Map/FormData/NodeList 等の iterable に欲しい機能」と再定義し理解を促進。
- 設計は Array 踏襲(名前 `includes`、SameValueZero、skip 数の第 2 引数)。第 2 引数は `drop` で代替可だが、Array との互換と negative index 不可で既に乖離する点から最終的に残す結論に収束。NaN/負数/非整数は throw。
- EAO の Python `in` 構文案は KG・MF が「既存の `in` 演算子の意味を変えられず、全 `in` を遅くする」として却下。SFC の composite 対応は composites 提案側の equality に従う(ACE: interned なら追加 spec 不要)。
- 結論: Stage 1・2 を取得し、reviewer(RGN ほか)と editor(JHD・RBN)の sign-off により Stage 2.7 まで進んだ。
