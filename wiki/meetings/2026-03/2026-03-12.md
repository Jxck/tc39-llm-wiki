# 2026-03-12 (113th TC39 Meeting, Day 3)

## First-class Protocols Update (continuation)

- Slides: [link](https://tc39.es/proposal-first-class-protocols/slides/2026-03/)
- LVU が前日の発表内容を再度通し、現行設計(members は `requires` で明示、computed property name で外部 Symbol も利用可、protocol は immutable)と open questions を確認した。
- EAO は class の多重継承への転用を問うたが、protocol は prototype chain を使わず member を copy する仕組みであり、それは mix-ins proposal の領域だとして champions は否定的。
- Symbol-first な設計に対し PFC/EAO が「string key に見えるのに Symbol」という直感に反する点を懸念。SFC は既存 protocol の多くは string ベースであり Symbol-first に懐疑的、RBN は C# 風の `extends ParentClass, ProtocolA,...` 記法を提案した。
- 命名は TypeScript チームが `interface` 呼称に反対した経緯から `protocol` を維持。
- Conclusion: stage 遷移はなし。Stage 2 に向けた設計を固めるためのフィードバック収集にとどまった。

## test262 coverage strategies

- test262 は built-in algorithm の分岐網羅(特に入力検証ステップの実行順序)が不十分で、既存のテストも煩雑だという問題提起。
- RGN は abrupt completion を引き起こす入力を early から late へ順次緩和する proof of concept を提示し、proxy ベースの probing で効率的な分岐網羅を試みた。
- 議論では dynamic evaluation ではなく code generation で行う案も出たが、test262 への contribution/review が高摩擦になる懸念が示された。
- proxy ベースの probing は最適化を回避するため非 proxy 経路の網羅も別途必要になりうる点が指摘された。
- Conclusion: 今後の進展は test262 の PR という形で行う(top-level algorithm ごとに throws 系テストを集約する方向)。

## TypedArray concat

- Slides: [link](https://github.com/tc39/proposal-typedarray-concat/blob/tc39-plenary-march-2026/slides-export.pdf)
- JSL は stage 前進を求めず、iterable 対応の是非について committee の温度感を測った。命名は `concat` から `from list` 系へ傾いていると述べた。
- iterable は full consume の必要や型不一致時の coerce/throw など複雑性が大きく、ecosystem の既存ユースケース(Node の buffer concat 等)では不要との意見が RGN/PHE から出た。
- MF は「iterable と array-like は同じ理由(利便性)で検討されているので採否は揃えるべき」とし、JSL の「最初の版は TypedArray と ArrayBuffer のみ」という説明を受けて不採用に同意。KM は `Symbol.isConcatSpreadable` 非対応を強く要望した。
- Conclusion: iterable は採用しない。関数名は `concat` から改名する。stage 前進の ask は May に持ち越し。

## TypedArray findWithin continuation

- Slides: [link](https://github.com/tc39/proposal-typedarray-findwithin/blob/tc39-plenary-march-2026/slides-export.pdf)
- JSL は concat 同様に stage を求めず、`contains` の削除や iterable サポートの除去(concat の先例に追随)を報告した。
- open question は新関数の命名、もしくは `indexOf` / `lastIndexOf` の overload で済ませるか。iterable 非対応により coercion がなくなるため、唯一の挙動変化は TypedArray を渡したとき従来の `-1` ではなく index を返す点のみ。
- KG/MF は overload を支持(Node の buffer indexOf に長年の先例あり)。一方 RGN/NRO は web scale での後方互換リスクを懸念し、専用メソッドを望み、Stage 2 で browser に use counter を入れて確認すべきと主張した。
- Conclusion: iterable は concat 同様に不採用。overload か新関数かは結論に至らず、committee は新関数寄り。May に再度持ち帰る。

## Thenable Curtailment for Stage 2

- Slides: [link](https://docs.google.com/presentation/d/1i0jY_e-A3rkd9256VzirJdPVtHInYNGjtYq_KrTBMuo/edit)
- 任意の `then` プロパティを持つ thenable が promise resolution で特別扱いされ、繰り返しセキュリティバグの原因になる問題への対処。MAG は user code 実行の可能性がある場合のみ resolution を 1 tick 遅延させる 'Safe Resolve'(名称未定)を提案した。
- 実験として全 C++ promise resolution をこの経路に通し web platform tests を実行、ほぼ全て pass(失敗は resolution timing を観測する種類で削除/調整可能)。web 互換性の見込みは高いとした。
- open questions: userland への公開可否とその命名。MM は新 state を導入せず spec 内部 promise へ転送する書き換え案を提示、KG は resolve 関数の第二引数で safe resolve 挙動を切り替える案を出した。MF は predicate の公開にも関心を示した。
- Conclusion: Stage 2 に到達(KG/PFC らの support、objection なし)。MAG は userland 公開(おそらく resolve 関数の第二引数経由)を引き続き検討する。

## Iterator Includes

- 直前に承認された `iterator.includes` の spec text に、第二引数に非整数を渡すと receiver を close せずに throw するバグがあり、その修正 PR を提示した。
- iterator は consumer 側のエラー状態(不正引数を含む)でも close されるべきという convention に従っていなかった点の修正。
- Conclusion: PR は承認(複数の approval、objection なし)。

## Towards Structured Concurrency

- Slides: [link](https://docs.google.com/presentation/d/1vYh__pmW0QuIhxz9jPQbUXdkIrzD5p7Wjl6oTdKSBZI/edit?usp=sharing)
- 具体的な proposal ではなく、言語が向かいうる方向性のビジョン提示。KG は structured concurrency(concurrency を scope に束縛し、scope は子 task の完了/shutdown まで抜けず、あるタスクのエラーが他タスクを cancel する)の概念を紹介した。
- これは structured programming の類推に基づく約 10 年来のアイデアで、他言語では議論が進んでいるが TC39 ではほとんど扱われてこなかったとした。
- Conclusion: AbortController / AbortSignal を独立して動機づけられる拡張として積み重ね、新しい `using` 構文を多用して structured concurrency への道筋を作れるという方向性に committee から概ね支持。Safari は AbortController ベースの cancellation のみに commit し、別種の cancellation は導入しない方針。
