# 2026-03-11 (113th TC39 Meeting, Day 2)

## Temporal for Stage 4
- Slides: [link](https://ptomato.name/talks/tc39-2026-03/)
- 提案ページ: [Temporal](../../proposals/temporal.md)
- PFC が Temporal の Stage 4 を要求。Stage 3 入りは 2021 年(Stage 2.7 導入前)で、長い Stage 3 期間に polyfill や実装からのフィードバックを得て改善してきた経緯を振り返った。
- Stage 4 の入口要件を満たす状況を提示。test262 に約 5,000 件(全体の約 10%)のテストがあり、Firefox/V8 系に加え Ladybird も約 100% 適合、unflagged で複数実装が出荷済み。ECMA-262/ECMA-402 双方に PR があり editor も確認済み。
- spec text の公開形態について、(1) ECMA-262 にそのまま含める、(2) 別ページ(ECMA-262-2)にする、(3) 別標準(ECMA-4xx)にして専用 TG と editor group を持たせる、の 3 案を MF が提示。委員会の大勢は「当面は ECMA-262 内に統合のまま」を支持し、専用 TG 化にも消極的だった。
- 結論: Temporal は Stage 4 に進んだ。当面は別 spec 文書化も専用 TG 化も行わず、1 年以内に前提が維持されているか再確認する(NRO が follow-up を担当)。

## Import Text for stage 2.7 or 3
- Slides: [link](https://docs.google.com/presentation/d/1mbkpHjqihHBx6rp07Snb8uB8XevJk7jz2xPSyQppSjE/edit?usp=sharing)
- EAO が import attribute の新しい `type: "text"` 値(ファイルを文字列として import)について、Stage 2.7 とその先の advancement を要求。spec の大半は HTML/Fetch/CSP 側にあり、test262 と WPT に PR がある。Bun では無条件、Deno では `--unstable-raw-imports` で利用可能。
- 唯一の未決事項は PR #10 で、`CreateTextModule` AO の入力を "should represent textual data" な string 値と記述する変更。non-textual データ(WASM/JPEG など)を text として import する扱いをめぐり SFC が「good strings / bad strings」の観点でより強い制約(`must` や human-readable 定義)を主張したが、NRO・RGN・KM らは反対し、その挙動定義は HTML 側の範疇という整理になった。
- MF の editorial 提案を反映し PR #10 を merge。stage 2.7 reviewer の JHD も承認した。
- 結論: PR #10 を merge のうえ、Stage 2.7 を経て Stage 3 へ連続して advance した。

## Intl Energy Units Stage 1 Update
- Slides: [link](https://docs.google.com/presentation/d/1iwknifQLEIA42KIoveoNLrRZlhEitZQ_JTKADP20rMA/edit?slide=id.p#slide=id.p)
- SFC が ECMA-402 の現行 45 単位がどの「usage(用途)」に基づき選定されたかを振り返り、energy units 提案を「energy usages」として再構成。対象は電力料金・ユーティリティダッシュボード・EV 充電・太陽光発電など(必要単位は kilowatt-hour / watt / kilowatt)。
- 科学用途・産業規模・自動車馬力・天然ガス暖房・電池/小型機器・栄養などは明確に out of scope と整理。MM・JSC からは Kelvin や mmHg など除外単位や SI 系完全網羅の是非について質問があり、SFC は「completeness は非目標、usage 単位で判断」と回答。
- 委員会からは今後 usage を追加する際の定量的な選定基準の整備を望む声が出た。全 locale での単位確認(megajoule 利用地域など)の CLDR 調査が未完のため、今回 Stage 2 は要求せず。
- 結論: 将来会合で Stage 2 を要求予定。

## Error code property for Stage 1, 2, or 2.7
- Slides: [link](https://github.com/jasnell/proposal-error-code-property/blob/main/slides-export.pdf)
- JSL が Error の options bag に `code`(任意の値、慣例は string)を渡し instance property として設置する提案を提示。`cause` と同様の薄い syntactic sugar で、Node.js が 2017 年から導入し Axios/AWS SDK/Deno/Bun 等に広まった既存パターンを標準化する狙い。DOMException の既存 `code` とは衝突しないと説明。
- MM が「spec が throw する各 error にも code を付与する」follow-on の可能性を提起。PFC・KM は、観測可能性が増えて実装のステップ並べ替えの自由が失われる懸念、spec 全体に code を割り当てる作業量の膨大さを指摘。EAO・WH は namespace の land grab を懸念し、spec-mandated code を追うか否かを早期 stage で判断すべきと主張。
- ACE は structuredClone への `code` 反映も必要(cause と同様)と指摘。JGT は標準的な error 情報の置き場として支持しつつ、code の過信や error 型の少なさにも言及。JHD は reviewer に立候補し namespace 問題は実害が薄いと述べた。
- 結論: Stage 1 承認(JSL が当夜 tc39 org へ移管)。言語側で code を定義するか否か・その方法は継続議論とする。

## First-class Protocols Update
- Slides: [link](https://tc39.es/proposal-first-class-protocols/slides/2026-03/)
- LVU が、2017 年に MF が開始し長く停滞していた本提案を、新 co-champion(LVU・JHD)体制で再起動した update を実施。stage advancement は求めず、現状・2018 年からの変更点・未解決の設計課題への committee feedback を得るのが目的(Stage 2 は将来会合で要求予定)。
- 動機は、言語内に散在する ad hoc な protocol(iterator・thenable・toPrimitive 等)を reify し、author が自前の protocol/interface を symbol ベースで宣言・introspect・compose できるようにすること。requirements だけでなく provided members も扱う。DOM の EventTarget や web components の各種 capability を protocol 化する用途も提示。
- 委員会 feedback: MM は protocol の immutability・branding でなく property test での判定・生成 symbol を registered symbol にしないことを重視。ACE・DRR は class head の `implements` が type-stripping ツールに与える影響や、従来 runtime 挙動を持たなかった `implements` の意味変更による混乱を懸念。SHS・OFR は生成 symbol member を識別子名で書く点を懸念。DRR は提案を 3 つ(symbol 群の一括生成 / プロパティ群のコピー / interface 充足チェック)に分割する案を提示。
- 結論: 議論は時間内に完結せず継続扱い(本日は stage 遷移なし)。
