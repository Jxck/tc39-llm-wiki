# 2026-05-19 (114th TC39 Meeting, Day 1)

## Needs-consensus PR: use Numbers instead of reals as counters in Iterator builtins (#3776)

- Slides: [link](https://docs.google.com/presentation/d/1WVjZyIAjTPl3TDV6X2cxNp1n-bzulIy_XWyKOsSq7i4)
- iterator helper の callback に渡す counter を、spec 上の任意精度整数から float64 相当の数え方に変更する needs-consensus PR。`Iterator.prototype.includes` の実装時に JSC が発見した問題が発端。
- 対象は counter を持つ 7 つの iterator helper に加え、`Iterator.prototype.take` / `Iterator.prototype.drop`。take/drop は 2^53 を超える大きな有限値に対し即座に RangeError を投げる。
- どの実装も任意精度 int を使っておらず float64 で実装しているのが実態で、MF はこの変更自体は好まないが実装現実に合わせるべきとした。BigInt 化は DLM/OFR/PST が性能面で反対。
- 結論: PR を merge することで consensus 達成。

## Needs-consensus PR: Support BigInt coercion of integers expressed as exponential-notation strings (#3857)

- 指数表記の文字列(例 `4.2000e+4`)が整数を表す場合に `BigInt` constructor で受理できるようにする needs-consensus PR。現状は Number constructor は受理するが BigInt は SyntaxError になる非対称を解消する。Amount 提案が発見の発端。
- BigInt literal の文法変更ではなく built-in 関数の挙動変更。指数表記が precision を表すか(`1.00` を整数とみなしてよいか)という未解決の論点を OFR/CLA が提起。
- needs-consensus PR とするか stage process に乗せるかで意見が割れ、MF/OFR は staging をやや選好。RGN は最も保守的に一度 stage one を要求したが、agenda 追加が遅かったこともあり結論を見送り。
- 結論: needs-consensus PR として次回会合で改めて議論する。

## `Atomics.pause` for Stage 4

- proposal: [proposal-atomics-microwait](https://github.com/tc39/proposal-atomics-microwait)
- KM が SYG に代わり `Atomics.pause` の Stage 4 を要求。spin lock で suspend 前に短く spin する際の engine hook を提供する(x86 の pause、ARM の ISB 相当)。
- 同時に、どの実装も使っていない optional な pause 回数 parameter を削除する normative change を要求。将来 backoff 制御に使えなくなる懸念を RBN が示すも、NRO/WH は parameter があっても挙動が無視される以上効果はないと反論。
- RBN は懸念を blocking とはせず。WH は将来 parameter に意味を持たせる余地を残すため今は含めるべきでないとした。
- 結論: normative change(parameter 削除)を受理し、`Atomics.pause` が Stage 4 に到達。

## RegExp Buffer Boundaries for Stage 2.7 or 3

- Slides: [link](https://1drv.ms/p/c/934f1675ed4c1638/IQBpJAD8CJutTpZ3x70Mr8C6AQ6Z26p2_nowaDED0_XQfRY?e=FLSMPe)

## `export defer` Stage 2 status update

- Slides: [link](https://docs.google.com/presentation/d/1uYgMUqz5bZT3wwPARnyq7GrJlWFesIrmoCJA1iDH3W8)
- NRO と新 champion CLA が `export defer`(deferred re-exports)の Stage 2 update を提示。`import defer` と同じ `%Promise.all%` 経由の Promise leak を安全な PerformPromiseAll AO で修正、`export * from` との相互作用や namespace import 経由の binding 検証バグも修正(いずれも Stage 2 のため bug fix として consensus 不要)。
- `export defer` と `import defer *` の併用で tree shaking が失われる課題への解として、filtered namespaces を提案。`import { foo, bar } as obj` で必要な export だけを含む新規 namespace object を毎回生成(TG3 の非キャッシュ要件を満たす)。`export as`/`import defer`/dynamic import への展開も想定。`*` は従来通り全件 import を維持。
- bundler 関係者は tree shaking 容易化の観点でこの変更単体に関心。LCA も module system を完成させる方向として強く支持。dynamic import への適用は import の第 2 引数に新 property(filter または names)で対応予定。
- 結論: 原文に Speaker's Summary/Conclusion の記載なし。filtered namespaces の変更を merge し、dynamic import を含めた全体像を次回会合で再提示する方針。
