## Stable Formatting for Stage 2

- Slides: [link](https://docs.google.com/presentation/d/1Vnt2-ejFMpk3NstM2DS0TtSRIoDqm9jqE4siiuMe4qE/edit)
- Intl の出力が ICU/CLDR 依存で不安定なため、安定した locale-agnostic な整形を `zxx` locale 経由で提供する提案の Stage 2 ask。null/undefined の差異を避けるため、`zxx` 文字列を返す定数 `Intl.STABLE` を導入する。
- `Intl.DateTimeFormat` は Temporal 出力に合わせ 24 時間制、`Intl.NumberFormat` には SI 等を典拠とする short unit identifier を導入。`Intl.Collator`/`Intl.Segmenter` 等は安定挙動を定義できず本提案からは除外し、後続提案へ回す。
- `microsecond`(µ の Unicode 文字 U+00B5 か U+03BC か)と `mile-scandinavian`(`mil`/`smi`/`SE mil` 等)の short unit identifier に議論が集中。SFC は zxx が CLDR 的に安全か未確認である点を最大の課題とした。
- 結論: Stage 2 で承認。microsecond/mile-scandinavian の識別子や残課題は Stage 2 で継続。reviewer は RGN と DLM。

## Intl Sequence Units for Stage 1 or 2

- Slides: [link](https://tc39.es/proposal-intl-sequence-units/slides/stage_1_or_2.html)
- 「5 ft 11 in」のように複数単位を連ねて整形する機能を `Intl.NumberFormat` に追加する提案。現状は複数 formatter と ListFormat を手動合成する必要があり、符号処理など footgun が多い。
- `-and-` infix(CLDR / UTS 35 由来)で `foot-and-inch` のように単位を指定し、format には各 subunit のフィールドを持つ object を渡す。負数は最大単位にのみ符号を付け、中間単位は整数必須。許可される組み合わせは 6 つの sanctioned group に限定し locale 非依存。
- scalar 入力か object 入力かが大きな論点。JHD は scalar の 6.5 が feet か inch か直感的でないと指摘、EAO は fractionDigits 適用時の曖昧さから object を支持、WH は scalar なら浮動小数点誤差回避のため最小単位にすべきとした。
- 結論: Stage 1 と、object ベース設計での Stage 2 を獲得。scalar は将来再検討可能。reviewer は EAO と DLM。

## SetFunctionName refinement

- accessor 関数の `Function.prototype.toString` で get/set prefix を付けるか否かに関する PR。前日議論した normative 版(関数ごとの自由選択を禁止)は今会合には行き過ぎとされ、本 PR(#3874)は editorial 版に切り替えたもの。
- `[[InitialName]]` を無条件設定する normative optional step と、prefix 有無を実装定義とする legacy step の二段構成。prefix の有無は実装定義かつ uniform であるべきで、built-in 間で混在させるのは discouraged legacy 挙動と note 化。
- JHD は note の "should" を "must" にすべきと主張したが、RGN は note は normative たり得ず本変更は現状の明確化(editorial)に留まると整理。NRO も normative とみなすべきとしつつ editorial discretion を許容。
- 結論: PR #3874 は editor group の discretion に委ねる。normative な #3855 は将来会合で再提示。

## Amount for Stage 2

- Slides: [link](https://docs.google.com/presentation/d/1Wq4uT0-lx8A_rfoSQ9uy5cnH1WHZXVjRmfYuugqkjQo/edit?slide=id.p#slide=id.p)

- number/BigInt/numeric string に optional な unit を組み合わせる container 型 Amount の Stage 2 ask(再提出)。単位変換 `convertTo()` を機能として含める。
- 前回からの改善点として convertTo の精度向上、numeric string の指数表記化、no-unit 表現に `~` を採用(YAML の null 由来)。conversion math は CLDR の比率を mathematical value で扱う方向だが、実装が約 2,100 個の数値を保持する懸念もある。

- 結論: Stage 2 で承認。reviewer は WH と JHD。変換 math の精度改善を Stage 2 で継続。

## export all from for Stage 1

- Slides: [link](https://docs.google.com/presentation/d/1IKj6GdnvI25gbqq_GUH-LQZOyfp0W1NFWc0hD4LU4CU)
- `export * from` が `default` を再 export しないため、proxy module(CDN の版エイリアス等)を汎用的に書けない問題を解決する提案の Stage 1 ask。2016 年に Guy Bedford らが提示し当時 block された案の再考。
- champion(NRO)の preferred solution は `export * from` の挙動を変更し常に `default` も再 export すること。複数 module の `default` 衝突時は named export と同じく曖昧で import 時にエラーとなる。新構文の追加は問題空間に見合わないと判断。
- web 互換性は namespace import に `default` が増える点が懸念だが、ライブラリへの export 追加同様に低リスクと評価。JHX は 10 年前の block 理由(default の特殊性)を踏まえ慎重さを求めた。JHD は別途 `export default from` も championship する意向を表明。
- 結論: Stage 1 で consensus(speaker summary/conclusion は未記載)。CommonJS 相互運用への影響は ZTZ が調査予定。

## Module Scope Ceiling

- Slides: [link](https://github.com/endojs/proposal-module-global/tree/main/slides)
- 旧 module global 提案の status update。新 Global を作らず、module の lexical lookup が global scope に到達しないようにする surgical な変更へ方針転換。
- `InitializeEnvironment` で module environment record の `[[OuterEnv]]` を Realm の globalEnv ではなく `[[OuterEnv]]` が null の `ScopeCeiling`(CM 命名)に差し替える。`Object`/`Function` 等は同 Realm のものが引き続き利用可能。DSL(テストランナーの `describe`/`it` 注入)や supply-chain security(LavaMoat)が動機。
- JRL は HTML spec が global environment record に settings object を載せている点との統合を懸念。NRO は HTML が lexical scope でなく Realm 経由で global を取得するため整合し得ると補足。`OuterEnv` が spec fiction か(実装依存)が open question。
- 結論: status update のみ(stage 遷移なし)。`ScopeCeiling` の露出方法やブラウザ実装可否は継続調査。

## RegExp Buffer Boundaries Continuation

- Slides: [link](https://1drv.ms/p/c/934f1675ed4c1638/IQBpJAD8CJutTpZ3x70Mr8C6AQ6Z26p2_nowaDED0_XQfRY?e=FLSMPe)
- Day 1 で `\Z` を CRLF または LF 相当と説明したのは簡略化であり誤りだったとの訂正。Stage 1 当時の提案は `\R`(LineTerminatorSequence)と同じ照合で、multiline `$`(CRLF に非対応)とは異なる。
- `\Z` は zero-width assertion で、buffer 末尾の直前に optional な line terminator sequence(CRLF/LF/CR/LS/PS)を許す。Java/.NET は既定で CRLF に対応(issue #20)。等価グループは non-capturing にすべきと CPC が指摘。
- RBN は本訂正に対する consensus を求めるか、`\Z` を切り離して別提案化し `\A`/`\z` を先行させるかの二案を提示し、まず consensus を求めた。
- 結論: `\Z` の訂正定義 `(?=(?:\r\n|\n|\r|\u2028|\u2029)?(?-m:$))` で consensus。翌日の Stage 3 advancement に向け reviewer を募集。

## AsyncContext (Stage 2) web integration status update

- Slides: [link](https://docs.google.com/presentation/d/1Q3wGg0YbYXio6lO3fzbo2gKJFiTb1t4Z1W4313CHZ3I)
- async コードフローを通じて値を暗黙伝播する AsyncContext の web 統合 status update。TC39 側に変更はなく、複雑さの大半は膨大な web async API との統合にある。
- events が最難関。registration context を廃し dispatch context のみを扱う方針へ変更し、WHATWG で Stage 1 化。伝播可否を 5 つの原則で規定(EventTarget 自体は同期で非干渉、外部要因のリスナは empty context、同一対象や生成元 object のイベントは伝播等)。
- ABO が原則を web spec(WebIDL の extended attribute、HTML、IndexedDB 等)へ適用する PR 群を作成中。RGN は冒頭例の明快さに比べ伝播規則が複雑で、lexical scoping のように一枚で説明できる形を求めた。
- 結論: 伝播規則は lexical scoping 並みに単純であるべきとのフィードバック。明快に説明できる awkward な挙動を、説明困難な edge case より優先すべき。特定 web API の専門知識を持つ協力者を募集。

## Source Phase Imports, Request for Consensus for Normative Change: Ambiguous re-export handling for Source Phase Imports

- Slides: [link](https://docs.google.com/presentation/d/1h7tHdRqdkR0s_x6TOiSlOIH-wA17cN7Hz0BLjeuLhqc)
- source phase imports の binding が複数の star re-export で同一に解決される際、namespace 等と同様に dedup して throw しないようにする normative change の consensus 確認(issue #75、PR #76、ecma262 #3775)。99.9% の開発者には無影響な edge case。
- source phase import は import 宣言が新 binding を導入しない原則に従い、同一 module source への複数経路は同じ binding とみなせるかが論点。LCA は host hook が同一入力に同一 source を返す契約から、namespace と同様 slot 相当の binding とみなせると主張。GB は abstract module record に module source 用の slot が存在すると確認。
- NRO は import star の場合と異なり統一すべき二形態が無く内部整合性は弱いが、他の import 種別との一貫性のために良いとした。
- 結論: source phase imports を曖昧再 export の dedup に参加させる normative change で consensus。現行の rebase で自然にこの挙動となるため明示的な spec 変更は不要。

## Thenable Curtailment status update

- Slides: [link](https://docs.google.com/presentation/d/1SW8xSjuWqrdix_MOE3O-ZJcR-_XgO4z2ajhJInz0IwE/view)
- thenable(`then` を持つ object)が Promise 解決時に特別扱いされ、`Object.prototype.then` 定義による security exploit が継続発生している問題への対処提案の update。前回以降もさらに脆弱性が増加。
- user code を起動せず Promise を解決する abstract operation(仮称 SafeResolve)を新設する方向。この capability を user に露出すべきかを temp check した結果、probably 12 / no 6 / yes 3。
- 露出方法は resolve 関数への第 2 引数案が現状の best idea だが、LCA は Boolean coercion の互換リスクを懸念し well-known Symbol や `resolve.safe` を提案。`Object.prototype.then` の exotic 化(KM/MF)や、user code 込みで解決する JRL 案(Svelte は破壊するが security 改善)も議論。
- 結論: 本提案は次会合での Stage 2.7 を目指し MAG が spec text を継続作成(JHD・Justin Ridgewell が reviewer)。user 露出は本提案の迅速化のため別提案へ分離。露出反対者は repository で理由を共有するよう招請。
