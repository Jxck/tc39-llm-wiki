# 2025-11-19 (Day Two)

## Async iterator helpers recap/update/polyfill

Kevin Gibbons (KG) より、非同期イテレータヘルパー(`AsyncIterator.prototype.map` 等)および同期イテレータヘルパーの実装状況のリキャップが行われました。ポリフィルの提供状況、各エンジンでの実装進捗、過去に指摘された仕様上のエッジケース(早期リターンやエラー伝播)の対応状況が共有されました。本セッションは情報共有が主旨であり、新たな Stage 昇格の要求はありませんでした。

## Iterator Join for stage 1, 2, or 2.7

- 提案ページ: [Iterator Join](../../proposals/iterator-join.md)

イテレータの要素を区切り文字で連結する `Iterator.prototype.join()` の提案です。配列の `Array.prototype.join` に相当する機能をイテレータに対して提供するもので、Stage 2.7 への前進が議題となりました。API 設計や無限イテレータに対する挙動について質疑がありましたが、概ね支持を集めました。

Conclusion: Stage 2.7 への昇格が承認されました。

## Composites comparator choice

Ashley Claymore (ACE) を中心に、Composites(構造的等価性を持つ値)における比較子(comparator)の選択方針についての議論が行われました。値の内部化(interning)の方式や、漏えいした WeakMap への挿入時に例外を投げるべきかなど、設計上の選択肢について TCQ による温度感の確認(temperature check)が2回実施され、いずれも内部化方式および「漏えいした WeakMap への挿入時は例外を投げる」方針が支持されました。本セッションでは正式な Stage 昇格の要求はなく、設計の方向性を確認する位置づけでした。

## Concurrency Control stage 1 update

並行処理の制御(ロックやセマフォに類する仕組み)を JavaScript に導入するための提案の Stage 1 進捗報告です。ユースケースの整理や、Web Locks API など既存の関連仕様との関係について説明があり、今後の設計方針についての委員会からのフィードバックが求められました。Stage 昇格の要求は無く、現行 Stage 1 のまま継続審議となりました。

## `Object.propertyCount` for Stage 2

オブジェクトの列挙可能プロパティ数を取得する `Object.propertyCount`、および文字列キーのみの数を返す `Object.keysLength` の提案です。議論の結果、2つの API を分離する方針となり、`Object.keysLength` についてのみ Stage 2 への前進が支持されました。`Object.propertyCount` は仕様設計上の懸念(Proxy やレシーバとの相互作用)が残るため、Stage 1 のまま継続審議となりました。

Conclusion: `Object.keysLength` は Stage 2 へ前進し、`Object.propertyCount` は Stage 1 のまま継続となりました。

## WHATWG Stages

WHATWG など他の標準化団体が定義する仕様提案について、TC39 内での Stage 概念の適用方法・連携プロセスについての議論です。クロスオーガニゼーションでの提案管理における手続き上の整理が主な内容で、特定の提案の Stage 昇格を伴うものではありませんでした。

## Spec-defined or Implementation-defined limits

ECMAScript 仕様における各種上限値(文字列長、配列長など)を仕様で定義すべきか、実装依存とすべきかについての方針議論です。既存の実装間差異と相互運用性の問題が共有され、今後どの範囲を仕様で明文化すべきかについて委員会の意見が求められました。具体的な Stage 昇格や Normative PR の提起はありませんでした。

## Module-declaration like proposals in other areas of the web

モジュール宣言(`import`/`export`)に類似した構文パターンが、Web の他の領域(HTML の `<script type=module>` 拡張や CSS の `@import` など)でどのように扱われるべきかについての方針議論です。TC39 が定義する構文と他団体の仕様との整合性を保つための調整が主旨で、特定の提案への Stage 判断は行われませんでした。

## Comparisons

値同士の比較演算(`Object.is` を拡張するような深い比較)についての提案の進捗報告です。API 設計について複数の選択肢が議論されましたが、委員会内での合意形成には至らず、引き続き Stage 0 のまま設計を練り直す方針となりました。

Conclusion: Stage 昇格は無く、Stage 0 のまま継続審議となりました。
