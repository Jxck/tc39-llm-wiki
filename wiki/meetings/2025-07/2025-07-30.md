USA(Ujjwal Sharma)が非 ISO8601 カレンダーの era code・epoch year・年の加算制約などを定義する提案を Stage 2.7 に向けて説明した。EAO と SFC がレビューを完了し概ね支持を表明したが、PR69(prose をアルゴリズムステップに変換)の扱いと Ecma-402 editor(RGN)の最終サインオフが未了だった。議論の末、SFC は PR69 を editorial と理解した上で容認し、editor レビューを条件として Stage 2.7 への advancement で consensus が得られた。

## Module Import Hook and new Global for Stage 1

- Slides: [link](https://github.com/endojs/proposal-new-global/blob/main/slides/slides.pdf)

KKL と ZTZ が、realm から global を分離する新しい `Global` constructor と import hook の提案を Stage 1 に向けて発表した。Compartment/サンドボックス用途や AI 生成コードの分離、test runner・他ホストのエミュレーションなどが動機として挙げられたが、KG・MAG・JHD らから「サンドボックスに見えるが実際はサンドボックスではない」という強い懸念、HTML 仕様が realm と global を一対一と仮定している実装上の懸念(MAG)、import map との重複(KG)など多くの異論が出た。時間切れとなり、問題文(problem statement)を見直す必要があるという結論で当日はテーブル送りとなった(継続審議は同日後半の Continuation セッションへ)。

## Import Buffer for Stage 1

- Slides: [link](https://proposal-import-buffer.vercel.app/)

STY(Vercel)が import attributes を使い任意のバイト列をモジュールとしてインポートする `import buffer` 提案を発表した。当初 `ArrayBuffer` での提案だったが、NRO・JSL・KM・MM らの支持により `Uint8Array`(immutable な `ArrayBuffer` を裏に持つ)へ変更され、属性名も `type: "buffer"` から `type: "bytes"` へ変更することで合意した。仕様の主要設計が固まっていることが確認され、Stage 1 だけでなく Stage 2 まで consensus が得られた(`Import Bytes has Stage 2`)。EAO が提案した `type: "text"` の追加は別提案として持ち越しとなった。

## Continuation: Module Import Hook and new Global for Stage 1

- Slides: [link](https://github.com/endojs/proposal-new-global/blob/main/slides/stage1.pdf)

ZTZ が改訂した問題文(「同一 realm 内の新しい Global スコープの文脈でモジュールとその依存関係を評価する方法」)を提示し、提案名を暫定的に `proposal-module-global` に変更すると発表した。realm と global の一対一関係が崩せない場合は提案自体が成立しないことが JHD・KG・KKL の間で確認されたが、実現可能性については楽観的な見通しが共有された。ModuleSource の import hook を worker 間でどう扱うかなど Stage 2 以降の論点も挙がったが、結論として新しい問題文の下で Stage 1 の consensus が得られた。

## Continuation: Non-extensible applies to private

- Slides: [link](https://github.com/tc39/proposal-nonextensible-applies-to-private/blob/main/no-stamping-talks/non-extensible-applies-to-private-update.pdf)

OFR が、Chrome Status のグラフに見られた 6 月の使用率上昇は httparchive.org のインデックス対象月によるノイズ(絶対件数は 6〜10 件程度)であり実質的な懸念材料ではないと説明した。MM は test262 の整備が唯一残る課題であるとし、JRL・NRO とのバンドラー実装(Babel 等)に関する確認を経て、次回会合での advancement(Stage 3 相当)に向けてブロッカーは無いという結論となった。

## Continuation: [~~Measure~~Amount](https://github.com/tc39/proposal-measure) for Stage 2

- Slides: [link](https://docs.google.com/presentation/d/1my6X1ODDckzJmtcWcFI9hRF_I06Z4RQwrq81lbo8wPM/edit?usp=sharing)
- 提案ページ: [Amount](../../proposals/amount.md)

JMN・SFC・EAO・WH・NRO らが、前日の `IntlMathematicalValue` の上限・下限の扱いについて続きの議論を行った。WH は、実装上限を強制する責務をこの AO から外し、巨大・極小値は指数表記で出力すべきと提案し、SFC も賛同した。仕様で上限を固定すべきか実装依存にすべきか(BigInt 同様)については未決着のまま、SFC が今後の会合で PR を出して継続検討する方針となった。Stage 変更には至らず、Speaker's Summary / Conclusion は具体的記述なし(プレースホルダ)のまま記録されている。
