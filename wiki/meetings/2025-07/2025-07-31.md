## `Object.propertyCount` for Stage 2

RBR が前回 Stage 1 で合意した `Object.propertyCount` を、非インデックス文字列プロパティの取得やスパース配列検出といった要素を切り出して別提案に分離し、スコープを縮小する形で再提案した。委員会では enumerable/non-enumerable/all の三状態オプションの設計や、options bag か個別メソッドかという API 形状で議論が紛糾し、KG・MF らは既存の `Object.keys`/`getOwnPropertyNames` の体系を埋めるための新規 API追加に懐疑的だった。さらに MM は本提案が後続の3提案(`Array.isSparse`、`Array.getNonIndexStringProperties`、`Object.getOwnPropertySymbols` options)と密接に関連するとして、4件を一体として検討すべきと主張し、議論の先延ばしを求めた。

### Conclusion

MM と KG が Stage 2 への advancement に objection を表明し、Stage 2 には進めなかった。4提案全体の API 過多・問題領域の重複が主な懸念として残った。

## `Array.isSparse` for Stage 1

RBR が配列がスパース(穴あき)かどうかを判定する `Array.isSparse`(または `isDense`)を Stage 1 として提案。正当性とパフォーマンスの観点(`Object.keys().length` 同様、Node.js の `assert` 実装などで非効率な穴チェックが行われている)を動機としたが、実装者(KM、OFR)から V8 では場合により O(1) になりうるが JSC では常に O(n) であるとの指摘があり、エンジン依存の非決定的な性能特性を公開することへの懸念が強く出た。WH は「false が返っても穴が無いことを保証しない(array-like を考慮すると三値になる)」という API の極性問題を指摘し、`isDense` への改名や proxies・shape 変化(densify されても内部表現が変わらない)の問題も議論された。

### Conclusion

WH・KM が問題空間自体に解決策が存在するか疑問を呈し objection、ZTZ・OFR らの支持はあったが、Stage 1 へは進まなかった。今後ベンチマーク等で説得力のある問題提起があれば再提案の道はあるとされた。

## `Array.getNonIndexStringProperties` for Stage 1

- Slides: [link](https://github.com/BridgeAR/array-get-non-index-string-properties/blob/main/slides/07.2025%20-%20proposal-slides.pdf)

RBR が配列・TypedArray 上の非インデックス文字列プロパティ(例: 正規表現の `match`/`matchAll` が返す配列に付与される `index`、`groups` 等)を取得するヘルパーを Stage 1 として提案。MM は、インデックスプロパティを先頭に列挙する現行仕様のため非インデックスプロパティだけを取得する手段が無いことを強い動機として支持し、enumerable のみでなく non-enumerable も含めるべきと提案した。配列限定にするか `Reflect.ownKeys` 系に一般化するかや、proxy・array-like 対応は Stage 2 以降の課題として整理された。

### Conclusion

MM・NRO・USA らの強い支持があり、objection なく Stage 1 に到達した。

## `Object.getOwnPropertySymbols` options for Stage 1

- Slides: [link](https://github.com/BridgeAR/object-get-own-property-symbols-options/blob/main/slides/07.2025%20-%20proposal-slides.pdf)

RBR が `Object.getOwnPropertySymbols` に enumerable フィルタオプションを追加する提案。symbol キープロパティの大半は non-enumerable で隠蔽用途に使われるため、呼び出し側で enumerability を逐一チェックするオーバーヘッドを避けたいという動機。KG・KM は options bag ではなく既存の `Object.keys`/`getOwnPropertyNames`/`getOwnPropertySymbols` の行列を埋める専用メソッドにすべきと強く主張し、MM は API 全体の肥大化を懸念し、4提案を一体の問題として再検討すべきとの立場を改めて述べた。

### Conclusion

KG・KM・MF・MM らから幅広い支持があり、objection なく Stage 1 に到達した(KG は命名に難色を示しつつ問題領域自体には同意)。
